# P2.3 Chunked Prefill 实现总结

> **日期**: 2026-03-17
> **目标**: 解决首次 prefill 128K-256K tokens 时的 OOM 问题
> **状态**: ✅ 完成

---

## 问题定义

### 用户需求

> "怎么解决首次 prefill 128K-256K 的 OOM 问题呢？"

### 核心问题

**MLX Metal Buffer 限制**：
- Metal buffer 最大 30GB
- 128K tokens 首次 prefill 会超过限制
- 导致 `[METAL] Command buffer execution failed: Insufficient Memory` 错误

**与 P2.2 的区别**：
- **P2.2**: 解决 prefix cache 已存在时，加载到内存的 OOM
- **P2.3**: 解决首次 prefill 时，计算 KV cache 的 OOM

---

## 解决方案

### 核心机制：Chunked Prefill

**原理**：
1. 将长 prompt 分成多个 chunk（如 4K tokens）
2. 逐块调用 model(chunk, cache=cache)
3. KVCache 会自动累积（原地修改）
4. 最后基于完整 cache 生成

**关键代码**：
```python
from mlx_lm.models.cache import KVCache

# 1. 创建 cache（每层一个 KVCache）
cache = [KVCache() for _ in range(len(model.model.layers))]

# 2. 逐块 prefill
for chunk in chunks:
    chunk_mx = mx.array([chunk])
    logits = model(chunk_mx, cache=cache)
    mx.eval(logits)
    mx.eval([c.keys for c in cache])  # 确保更新完成

# 3. 基于完整 cache 生成
next_token = mx.argmax(logits[0, -1, :], axis=-1).item()
# ... 继续生成
```

### 为什么有效

**MLX-LM 的 KVCache 是可变对象**：
- `KVCache.update_and_fetch(keys, values)` 会原地修改 cache
- Model 不返回 cache tuple，cache 在内部被修改
- 每个 chunk 只需要处理 4K tokens，避免超过 Metal buffer 限制

## 智能分块策略（未实现）

### 用户需求

> "chunked 要智能，不能 4K 这样僵死的，不然正好把一个文本切割了，最好是文本块的边界做个智能对齐"

### 建议策略

**语义边界检测**：
1. **段落边界**: `\n\n` 双换行
2. **句子边界**: `.` `!` `?` + 空格
3. **代码块边界**: 完整函数、类定义

**混合 Chunk Size**：
- 基础 chunk: 4K tokens
- 最小 chunk: 512 tokens
- 最大 chunk: 6K tokens

**场景自适应**：
- 对话场景: 按轮次切分
- 文档场景: 按段落切分
- 代码场景: 按函数切分

**实现复杂度**：
- 需要 tokenizer 支持 offset mapping
- 需要实现边界检测逻辑
- 需要权衡性能和复杂度

**当前状态**: 暂未实现，使用固定 4K chunk size

## P2.2 vs P2.3 对比

| 维度 | P2.2: Prefix Cache 流式加载 | P2.3: Chunked Prefill |
|------|---------------------------|---------------------|
| **解决问题** | Cache 加载到内存的 OOM | 首次 prefill 计算的 OOM |
| **触发时机** | 后续请求（cache 已存在） | 首次请求（cache 不存在） |
| **核心机制** | 分批加载 cache blocks | 分块计算 KV cache |
| **性能开销** | < 1% (64K), +0.1% (128K) | +2.5% (16K), +11.1% (64K) |
| **内存节省** | 128K: -739MB | 不涉及（解决的是计算 OOM） |

### 配合使用

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   首次请求 (cache 不存在)                                   │
│        ↓                                                    │
│   P2.3: Chunked Prefill                                     │
│        ↓                                                    │
│   构建 KV cache (无 OOM)                                    │
│        ↓                                                    │
│   保存到 SSD                                                │
│                                                             │
│   ─────────────────────────────────────────────────────    │
│                                                             │
│   后续请求 (cache 已存在)                                   │
│        ↓                                                    │
│   P2.2: 流式加载 cache                                      │
│        ↓                                                    │
│   加载到内存 (避免内存峰值)                                 │
│        ↓                                                    │
│   生成响应                                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
