# P2.2: 流式分批加载设计 - 支持 128K-256K 超长上下文

**创建时间**: 2026-03-17
**优化目标**: 长上下文场景的 KV Cache 加载，支持 128K-256K tokens 而不内存崩溃
**预期收益**: -20% 加载延迟 + 内存安全

---

## 1. 问题定义

### 当前问题

**场景**: OpenClaw 多 agent，128K tokens 长上下文

**当前实现**:
```python
# prefix_cache.py:1475
loaded_blocks_dict = self.paged_ssd_cache.load_blocks_batch(
    block_hashes, max_workers=4  # 一次性加载所有 blocks
)
```

**问题**:
```
128K tokens = 512 blocks (每个 block 256 tokens)
每个 block ≈ 40MB (解压后)
总内存：512 × 40MB = 20.5GB

M4 Pro 48GB 内存：
- 系统占用：8GB
- 模型权重：15GB (30B Q5_K_M)
- 可用：25GB
- 加载 512 blocks：20.5GB → 💥 接近极限，容易 OOM
```

### 目标场景

| 场景 | Token 数 | Block 数 | 内存需求 | 当前状态 |
|------|---------|---------|---------|---------|
| 单 Agent 短对话 | 2K | 8 | 320MB | ✅ 安全 |
| 单 Agent 长对话 | 16K | 64 | 2.5GB | ⚠️ 可用 |
| OpenClaw 5 Agents | 64K | 256 | 10GB | ❌ 危险 |
| 超长上下文 | 128K | 512 | 20GB | 💥 崩溃 |
| 极限场景 | 256K | 1024 | 41GB | 💥💥 不可能 |

## 2. 优化方案：流式分批加载

### 核心思想

```
分批加载 + 异步预加载 + 内存保护

┌─────────────────────────────────────────────────────────┐
│         Streaming Batch Load Architecture               │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  用户请求 (512 blocks)                                   │
│       │                                                  │
│       ▼                                                  │
│  ┌─────────────────────────────────────────────────┐   │
│  │  Phase 1: 优先加载 (High Priority)               │   │
│  │  - Load blocks[0:16] (System prompt + 初始对话)  │   │
│  │  - 同步加载，立即返回                             │   │
│  │  - 内存：640MB                                    │   │
│  └─────────────────────────────────────────────────┘   │
│       │                                                  │
│       ▼                                                  │
│  ┌─────────────────────────────────────────────────┐   │
│  │  Phase 2: 后台预加载 (Background Prefetch)       │   │
│  │  - Async load blocks[16:32]                     │   │
│  │  - 边处理边加载                                  │   │
│  │  - 内存增量：+640MB                              │   │
│  └─────────────────────────────────────────────────┘   │
│       │                                                  │
│       ▼                                                  │
│  ┌─────────────────────────────────────────────────┐   │
│  │  Phase 3+: 流式加载 (Streaming)                  │   │
│  │  - 循环加载 blocks[32:48], [48:64], ...         │   │
│  │  - 加载一批，处理一批                             │   │
│  │  - 内存控制：最多 3 批在内存 (1.9GB)             │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  内存保护：                                              │
│  - 监控可用内存 (psutil)                                │
│  - 可用 < 5GB → 暂停预加载                              │
│  - 可用 < 2GB → 触发 LRU 驱逐                           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 关键参数

```python
# 分批配置
BATCH_SIZE_INITIAL = 16        # 首批加载 blocks (优先级最高)
BATCH_SIZE_STREAMING = 16      # 流式批次大小
MAX_CONCURRENT_BATCHES = 3     # 最多 3 批同时在内存

# 内存保护
MEMORY_SAFE_THRESHOLD_GB = 5   # 可用内存 < 5GB 时暂停预加载
MEMORY_CRITICAL_THRESHOLD_GB = 2  # 可用内存 < 2GB 时强制驱逐

# 总内存控制
MAX_TOTAL_MEMORY_MB = BATCH_SIZE_STREAMING * MAX_CONCURRENT_BATCHES * 40  # 1.9GB
```

### 加载策略

**优先级分级**:
1. **P0 (立即加载)**: 前 16 blocks
   - 包含 system prompt (通常 2-4 blocks)
   - 包含最近对话 (12-14 blocks)
2. **P1 (后台预加载)**: 第 17-32 blocks
   - 异步加载，不阻塞推理
3. **P2 (按需加载)**: 剩余 blocks
   - 流式加载，处理一批加载一批

## 3. 实现方案

### 3.1 修改 `prefix_cache.py` 加载逻辑

**当前代码** (prefix_cache.py:1455-1477):
```python
# 收集所有需要加载的 blocks
blocks_to_load = [...]  # 可能有 500+ blocks

# 一次性加载全部 → 危险！
loaded_blocks_dict = self.paged_ssd_cache.load_blocks_batch(
    block_hashes, max_workers=4

```

**优化后**:
```python
def _load_blocks_streaming(
    self,
    blocks_to_load: List[Tuple[int, Block, bytes]],
    batch_size_initial: int = 16,
    batch_size_streaming: int = 16,
) -> Dict[bytes, Any]:
    """
    流式分批加载 blocks，支持超长上下文。

    Args:
        blocks_to_load: 需要加载的 blocks 列表
        batch_size_initial: 首批加载大小（优先级最高）
        batch_size_streaming: 流式批次大小

    Returns:
        加载的 blocks 字典 {block_hash: block_data}
    """
    total_blocks = len(blocks_to_load)
    loaded_blocks_dict = {}

    # Phase 1: 优先加载首批 blocks（同步）
    initial_batch = blocks_to_load[:batch_size_initial]
    if initial_batch:
        logger.info(
            f"⚡ [Streaming Load Phase 1] Loading initial batch "
            f"({len(initial_batch)}/{total_blocks} blocks, "
            f"~{len(initial_batch) * 40}MB memory)"
        )

        initial_hashes = [bh for _, _, bh in initial_batch]
        initial_dict = self.paged_ssd_cache.load_blocks_batch(
            initial_hashes, max_workers=4
        )
        loaded_blocks_dict.update(initial_dict)

    # Phase 2+: 流式加载剩余 blocks（异步）
    remaining_blocks = blocks_to_load[batch_size_initial:]

    if remaining_blocks:
        logger.info(
            f"⚡ [Streaming Load Phase 2+] Starting streaming load "
            f"({len(remaining_blocks)} blocks, "
            f"batch_size={batch_size_streaming})"
        )

        # 启动异步预加载器
        from .async_prefetcher import AsyncPrefetcher
        from .prefetch_cache import PrefetchCache

        prefetcher = AsyncPrefetcher(num_workers=4)
        prefetcher.start()

        # 分批加载
        for i in range(0, len(remaining_blocks), batch_size_streaming):
            batch = remaining_blocks[i:i + batch_size_streaming]

            # 内存保护检查
            if not self._check_memory_safe():
                logger.warning(
                    f"⚠️ [Streaming Load] Memory critical, "
                    f"pausing at batch {i//batch_size_streaming + 1}"
                )
                break

            # 加载当前批次
            batch_hashes = [bh for _, _, bh in batch]
            batch_dict = self.paged_ssd_cache.load_blocks_batch(
                batch_hashes, max_workers=4
            )
            loaded_blocks_dict.update(batch_dict)

            logger.debug(
                f"⚡ [Streaming Load] Batch {i//batch_size_streaming + 1} loaded "
                f"({len(batch)} blocks, total {len(loaded_blocks_dict)}/{total_blocks})"
            )

        prefetcher.stop()

    return loaded_blocks_dict

def _check_memory_safe(self) -> bool:
    """检查内存是否安全，可用内存 > 5GB 返回 True"""
    try:
        import psutil
        mem = psutil.virtual_memory()
        available_gb = mem.available / (1024 ** 3)

        if available_gb < 2:
            logger.error(f"❌ Memory CRITICAL: {available_gb:.1f}GB available")
            return False
        elif available_gb < 5:
            logger.warning(f"⚠️ Memory LOW: {available_gb:.1f}GB available")
            return False
        else:
            logger.debug(f"✅ Memory OK: {available_gb:.1f}GB available")
            return True
    except ImportError:
        # psutil 不可用，假设安全
        logger.warning("psutil not available, skipping memory check")
        return True
```

## 4. 性能分析

### 内存使用对比

| 场景 | 当前实现 | 流式加载 | 节省 |
|------|---------|---------|------|
| 128K tokens (512 blocks) | 20.5GB 峰值 | 1.9GB 峰值 | **-91%** |
| 256K tokens (1024 blocks) | 41GB 峰值 💥 | 1.9GB 峰值 | **-95%** |

### TTFT 影响

**Baseline (一次性加载)**:
```
加载 512 blocks: 512 × 50ms / 4 workers = 6.4s
TTFT: 6.4s (用户等待)
```

**优化后 (流式加载)**:
```
首批 16 blocks: 16 × 50ms / 4 workers = 0.2s
后台预加载: 异步，不阻塞
TTFT: 0.2s (用户等待)

改进: 6.4s → 0.2s = -96.9% ✅
```

### 吞吐量影响

**分批加载会降低总吞吐量吗？**

❌ **不会**，因为：
1. 首批 16 blocks 足够开始推理（4K tokens）
2. 后续 blocks 边加载边处理，流水线并行
3. 总加载时间不变，只是分散到多个批次

## 6. 风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
